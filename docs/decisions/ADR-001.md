# ADR-001: Backend and Database

| Field      | Value                                                          |
|------------|----------------------------------------------------------------|
| **Status** | Proposed                                                       |
| **Author** | @EmaJasekova |
| **Date**   | 2026-03-26                                                     |
| **Decision** | Supabase (PostgreSQL + Auth) as the backend platform         |

## 1. Context

BadBuy needs persistent storage and user authentication. The product is built by three students in two months. We want to spend effort on product quality, not on operating servers, hand-rolling auth, or managing infrastructure.

Two decisions needed: **(A)** use a Backend-as-a-Service instead of a custom backend, and **(B)** pick a concrete provider.

## 2. Considered Options

### 2.1 Custom backend + self-managed database

**Verdict:** Rejected. Designing and operating auth, migrations, deployment, and backups is too much overhead for a two-month project.

### 2.2 Firebase

BaaS with strong mobile SDKs, but its primary data model is document-oriented. Our domain is naturally relational: users, hobbies, tracked items, suggestions. We want to filter and query in the database with SQL, not work around a document store.

**Verdict:** Rejected. Relational needs outweigh Firebase's strengths for this app.

### 2.3 Other managed databases (Neon, PlanetScale, Railway, Render)

Strong for hosting PostgreSQL, but authentication would be a separate concern requiring additional integration work.

**Verdict:** Rejected. No single provider matched our combination of relational storage and integrated auth.

### 2.4 Supabase

PostgreSQL with built-in auth, Row Level Security, a TypeScript client SDK, and an optional cron job runner - everything we need in one platform.

**Pros:**
- Relational model with joins, constraints, and rich filtering without denormalization workarounds
- Built-in auth (email/password, OAuth, session handling)
- Octavian has prior Supabase experience
- `pg_cron` covers the daily currency rate sync without a separate service
- Row Level Security enforces data isolation at the database level

**Cons:**
- Vendor dependency
- Must stay within free tier quotas

**Verdict:** Best fit for our stack, timeline, and team skills. Selected.

## 3. Decision

We use Supabase as the backend platform: PostgreSQL for structured data, Supabase Auth for user sessions.

## 4. Risks and Mitigation

### R1: RLS misconfiguration exposing or blocking data

|                |                                                                                                   |
|----------------|---------------------------------------------------------------------------------------------------|
| **Risk**       | Incorrectly configured Row Level Security policies expose one user's data to another, or block legitimate access. |
| **Likelihood** | Medium                                                                                            |
| **Impact**     | High                                                                                              |
| **Mitigation** | Model and test RLS policies early. Review policies in code review. Test with multiple user accounts before release. |

### R2: Auth integration bugs on device

|                |                                                                                                   |
|----------------|---------------------------------------------------------------------------------------------------|
| **Risk**       | Deep links, refresh tokens, or secure storage behave differently on iOS and Android.              |
| **Likelihood** | Medium                                                                                            |
| **Impact**     | Medium                                                                                            |
| **Mitigation** | Follow Supabase and Expo documentation. Centralize auth in one module. Test sign-in, sign-out, and session restore on real devices early. |

### R3: Free tier quota limits during demo

|                |                                                                                                   |
|----------------|---------------------------------------------------------------------------------------------------|
| **Risk**       | Usage spikes during the demo hit Supabase free tier limits.                                       |
| **Likelihood** | Low                                                                                               |
| **Impact**     | Low                                                                                               |
| **Mitigation** | Monitor usage in the Supabase dashboard. Optimize queries if needed. Upgrade tier only if necessary for the demo. |

## 5. Consequences

**Positive:**
- No custom server or separate auth service needed for the initial release.
- The whole team works with familiar relational SQL - no new paradigm to learn.
- `pg_cron` handles the currency rate sync job without introducing a separate service.
- Built-in auth reduces security risk compared to a from-scratch implementation.

**Negative:**
- Tied to Supabase's operational and pricing decisions.
- Free tier constraints require mindful query optimization and usage monitoring.
