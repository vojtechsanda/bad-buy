# ADR-003: Application Folder Structure

| Field        | Value                                                          |
| ------------ | -------------------------------------------------------------- |
| **Status**   | Accepted                                                       |
| **Author**   | @vojtechsanda                                                  |
| **Date**     | 2026-04-06                                                     |
| **Decision** | Hybrid Vertical Architecture (Feature-Based with Shared Layer) |

## 1. Context

We are building BadBuy as a React Native (Expo) application (ADR-002) with file-based routing (Expo Router), Zustand for state management, and Supabase as the backend (see ADR-001 ). The codebase needs a folder structure that supports a 2-3 person team working in parallel over a ~2 month timeline, with clear ownership boundaries and minimal merge conflicts.

### Constraints

- **Routing:** Expo Router enforces file-based routing — every file in `app/` becomes a route.
- **Feature interconnection:** High. The vault reads decisions created by the audit. The dashboard aggregates data from everywhere.
- **Backend:** Supabase Edge Functions for AI features (price table, alternatives), co-located in the repository.

## 2. Considered Options

### 2.1 Flat Horizontal (Group by Technical Role)

The traditional approach: top-level folders for `components/`, `hooks/`, `stores/`, `services/`, `types/`.

**Pros:**

- Simple to set up and understand on day one.
- Familiar pattern from simple web React projects.

**Cons:**

- **Scales poorly with feature count.** As BadBuy grows, `components/` becomes a dumping ground of 30+ files with no clear structure or ownership.
- **No ownership boundaries.** Two developers working on different features (e.g., audit and vault) will both touch `components/`, `hooks/`, and `stores/` — merge conflicts become frequent.
- **High coupling by default.** Nothing prevents a vault component from importing an audit-internal hook. Dependencies become invisible and super monolitic.

**Verdict:** Acceptable for solo developers or very small apps, but inappropriate for our team building product on a tight timeline.

### 2.2 Fully Vertical (Feature Slices Only)

Every feature is fully self-contained: its own components, hooks, store, services, types, and even its own UI primitives.

**Pros:**

- Maximum isolation. Features are independently deletable.

**Cons:**

- **BadBuy has high feature interconnection.** The `ConfettiBlast` component is needed by both the audit result screen and the Think Tank "unfreeze" flow. The `useCurrency` hook is needed everywhere. Fully vertical forces duplication of these shared concerns, or breaks the "no cross-feature imports" rule constantly.
- **Design system fragmentation.** UI primitives like `Button`, `Card`, and `Toggle` would either be duplicated across features or require an unofficial shared folder - defeating the purpose.
- **Overkill for our scale.** Fully vertical architectures is great in large teams or micro-frontend setups. For 3 developers, the overhead of strict isolation exceeds the benefit.

**Verdict:** Architecturally clean in theory, but creates a duplication problem that directly conflicts with BadBuy's interconnected feature set.

### 2.3 Hybrid Vertical (Feature Slices + Shared Layer) ✅ Selected

Each feature owns its domain-specific code (components, hooks, store, service, types), while genuinely cross-cutting concerns live in a `shared/` layer.

**Pros:**

- **Feature ownership is clear.** Each feature folder (audit, vault, think-tank, profile, streak, auth) is a self-contained vertical slice. A developer working on the vault touches `src/features/vault/` and nothing else for feature-specific work.
- **Shared code is explicit, not accidental.** The `shared/` layer contains only things that are genuinely used by 2+ features: the design system (`Button`, `Card`, `SliderSave`), infrastructure (`supabase.ts`, `edge-functions.ts`), cross-cutting hooks (`useCurrency`, `useHaptics`), and celebration components (`ConfettiBlast`).
- **Simple import rules prevent coupling.** Features import from `shared/` freely. Features can read another feature's `types.ts`. Features never import another feature's components, hooks or store. If two features need the same component, it moves to `shared/`.

**Cons:**

- Slightly more directories than the flat approach - initial setup requires a few more folders.
- The `shared/` boundary requires team discipline. Without the import rules, `shared/` can become a dumping ground similar to the flat structure.

**Verdict:** Best balance of separation, reuse, and usability for 3 person team building an interconnected product.

## 3. Decision

**We choose the Hybrid Vertical Architecture.**

The `app/` directory contains only Expo Router route files (screens). All application logic lives under `src/`, split into `features/` (vertical slices per domain) and `shared/` (cross-cutting design system, infrastructure, and utilities). The Supabase backend (migrations + Edge Functions) is co-located under `supabase/`.

### Feature Slice Structure

Each feature follows a consistent internal layout:

| File/Folder   | Purpose                                   |
| ------------- | ----------------------------------------- |
| `components/` | React components specific to the feature  |
| `hooks/`      | React hooks specific to the feature       |
| `store.ts`    | Zustand store for the feature's state     |
| `service.ts`  | Supabase calls for the feature            |
| `utils.ts`    | Pure utility functions (where applicable) |
| `types.ts`    | TypeScript types for the feature          |

## 4. Risks and Mitigation

### R1: Shared layer becomes a dumping ground

|                |                                                                                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Risk**       | Developers default to putting code in `shared/` instead of in feature folders, eroding the vertical architecture over time.                                         |
| **Likelihood** | Medium                                                                                                                                                              |
| **Impact**     | Medium                                                                                                                                                              |
| **Mitigation** | Enforce the import rule in code reviews: a component/hook only moves to `shared/` when a second feature needs it. Start in the feature folder, promote when needed. |

### R2: Route file confusion with Expo Router

|                |                                                                                                                                                                                                                        |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Risk**       | Accidentally placing non-route files in `app/`, creating unintended routes.                                                                                                                                            |
| **Likelihood** | Low                                                                                                                                                                                                                    |
| **Impact**     | Low (easily caught)                                                                                                                                                                                                    |
| **Mitigation** | Strict convention: `app/` contains only `_layout.tsx` files and screen files (`.tsx` that default-export a screen component). All logic, components, and hooks live in `src/`. Enforced in onboarding and code review. |

### R3: Feature folder overhead feels excessive early on

|                |                                                                                                                                                                                                                   |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Risk**       | In the first week, having this kind of separation feels like over-engineering.                                                                                                                                    |
| **Likelihood** | High                                                                                                                                                                                                              |
| **Impact**     | Low (cosmetic)                                                                                                                                                                                                    |
| **Mitigation** | This is expected and temporary. Features fill out quickly as screens are built. The alternative - starting flat and restructuring later - costs far more time mid-project than starting with the right structure. |

## 5. Consequences

**Positive:**

- Each developer can own a feature with minimal overlap and merge conflicts.
- Adding a new feature is a new folder under `features/`, not a refactor of existing code.
- The structure maps directly to the app's navigation (home, audit, think-tank, vault, profile), making it easy to locate any piece of code.
- The Supabase backend (migrations + Edge Functions) is co-located but independent - a teammate can work on the schema without touching any React Native code.

**Negative:**

- More initial boilerplate than a flat structure.
- Requires team alignment on the import rules - one undisciplined cross-feature import can set a bad precedent.
- The `shared/` layer still needs periodic review to ensure it contains only genuinely shared code.
